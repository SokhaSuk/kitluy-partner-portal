# Data model

The delivered browser domain is schema version 1. `src/data/domainSeed.js` declares every required
collection; `src/store/domainRepository.js` validates and persists the complete snapshot.

## Envelope and metadata

```js
{
  schemaVersion: 1,
  savedAt: "2026-07-13T00:00:00.000Z",
  data: {
    meta: {
      schemaVersion: 1,
      createdAt: "2026-07-13T00:00:00.000Z",
      updatedAt: "2026-07-13T00:00:00.000Z"
    },
    // required collections below
  }
}
```

Active storage is scoped as `kitluy.partner.domain.scope.<encoded scope>`, generated by
`domainStorageKeyForScope()`. Scope is `guest` or an immutable local-auth `accountId`; display email,
phone, and store name never select storage. `DOMAIN_STORAGE_KEY` (`kitluy.partner.domain`) is retained
as the pre-scope legacy key, not the current shared source of truth. Code/tests must import the key
helpers/constants rather than duplicate literals.

Imports accept a current-version envelope or bare snapshot, are limited to 5 MB, and validate the
complete object tree before explicit replacement.

Schema 1 is the first supported schema, so there is no historical migration chain yet. A future
breaking shape requires a version bump, explicit compatibility/migration logic, and fixtures/tests for
the prior version. Future-version snapshots are rejected.

## Required schema-v1 collections

| Area | Collections | Current purpose |
|---|---|---|
| Orders/customers | `orders`, `customers`, `customerNotes`, `customerPreferences` | Order/customer records, history, notes, and preferences |
| Communications | `threads` | Thread/message history, unread state, local outbox |
| Catalog/growth | `services`, `addons`, `priceHistory`, `promotions`, `marketingFlows`, `offers`, `adCampaigns` | Local services/prices and marketing configuration |
| Inventory | `inventoryItems`, `purchaseOrders`, `suppliers`, `adjustments`, `counts`, `stockMoves`, `inventorySettings` | Stock master records and audited workflows/movements |
| Production | `stations`, `workOrders`, `productionExceptions` | Work queues, capacity, and exception state |
| Team | `members`, `shifts`, `attendance`, `permissions` | Local staff/scheduling/access reference records |
| Store | `storeSettings`, `notifications`, `loyaltyMode`, `paymentOptions` | Store configuration and local preferences |
| Governance | `auditEvents` | Browser-local mutation history; not tamper-resistant compliance evidence |
| Integrations | `integrations`, `apiKeys`, `webhooks` | Local development status/placeholders/endpoints only |
| Data-driven pages | `genericState` | Persistent `{ toggles, actions }` for generic local controls |

All required keys are validated. Array collections must be arrays; object collections must be plain
objects. Unsafe prototype keys, malformed JSON, incomplete data, unsupported versions, and oversized
imports are rejected before replacement.

## Identity and references

New mutable records receive generated stable IDs. Existing seed shapes also retain human-readable
codes/names used by the reconstructed UI. Compatibility lookup helpers can resolve common references
by ID, code, name, label, type, or URL depending on the collection.

Preferred relations are stable IDs:

- order/customer and order/service;
- work order/order;
- customer note/customer;
- thread/customer or member;
- PO/supplier and PO line/inventory item;
- stock move/adjustment/count/source item;
- shift/attendance/member;
- audit event target `{ type, id }` in a future normalized model.

Do not add new name-only joins. A remote SQL/API model must normalize compatibility fields into
tenant-scoped foreign keys and constraints.

## Core invariants

### Orders

- KHR amounts are integer riel.
- Status changes append timeline and audit history.
- Dashboard, production, fulfillment, customer history, finance, and reports read the same order.
- Local operational/payment labels do not prove gateway settlement.

### Customers, notes, and threads

- Customer creation/notes use stable generated IDs and timestamps.
- Unread counts derive from canonical threads.
- Outgoing messages are local outbox entries; no external delivery state is fabricated.
- Complaint replies are generic local audit actions, not Telegram sends.

### Inventory and purchase orders

- `inventoryItem.quantity` is numeric and its `unit` is explicit; display stock strings are derived.
- Quantity parsing handles inventory labels and Unicode minus signs and rejects non-finite input.
- Adjustments append stock/audit movements rather than rewriting history.
- PO receipt uses the remaining quantity unless an explicit valid partial quantity is supplied.
- Count start/completion persists lifecycle state and posts variance movements.
- Duplicate/excess receipt/count behavior must remain idempotent/validated.
- Negative stock follows `inventorySettings`.

Mutation tests cover quantity parsing, partial PO remaining quantity, immutability, and audit append.
Rendered reconciliation/reload is still part of the unrun browser matrix.

### Catalog, marketing, and settings

- Service prices and campaign budgets are integer KHR.
- Add-on/flow/offer/settings toggles persist in the canonical snapshot.
- Promotion/campaign creation generates stable local records.
- Local activation does not mean POS/ad-provider publication.
- Price-history/audit evidence is append-oriented.

### Team and access

Members/shifts/attendance/permissions are local records. Production invitations, PIN hashes,
authorization, device sessions, and revocation must be enforced by a trusted backend; they are not
domain-snapshot security controls.

### Integrations

- `integrations.status` is explicitly a local development marker.
- `apiKeys` contain visible placeholders, not usable secrets.
- `webhooks` contain endpoint/event records, not signing secrets or delivery attempts.
- OAuth tokens, provider secrets, private payment data, and hosted-AI keys must never enter the domain
  snapshot.

### Generic state

`genericState.toggles` stores keyed local booleans. `genericState.actions` stores generated ID, key,
label, and timestamp for a persistent local interaction history. It is appropriate for data-driven
screens without typed invariants; it must not become storage for stock, orders, money, credentials,
or provider confirmation.

### Audit events

Domain mutations append IDs, event types, actors, targets, timestamps, and safe structured details.
This history helps local traceability but can be edited by the browser user. Never log passwords,
reset codes, provider tokens, full payment payloads, or unnecessary personal data.

## Money, quantity, and time

- Store KHR as integers such as `24800`, never formatted text or binary-floating currency.
- Store quantity as a finite number plus unit; format only at the view boundary.
- Store timestamps as ISO-8601; choose/display the store timezone explicitly.
- Define report date-range inclusivity and never parse localized display text back into domain truth.

## Local authentication stores

Authentication is intentionally outside the business backup/domain envelope:

- `kitluy.partner.local-auth.accounts.v1` in local storage;
- `kitluy.partner.local-auth.session.v1` in session or local storage based on remember-me;
- `kitluy.partner.local-auth.reset.v1` in session storage;
- legacy session key is cleaned during migration/initialization.

Password records use a random salt and `sha256-v1` hash. Reset challenges store a salted code hash,
expiry, and attempts. Hashes reduce plaintext exposure but do not make browser-controlled auth a
production security boundary.

The authenticated user exposes an immutable `accountId`, which is also the business-domain scope.
Session restoration, sign-in, and registration switch to that account scope; sign-out switches to
`guest`. Account identity and business data remain isolated even if display contact/store fields are
edited.

## Seed, backup, reset, and recovery

- The seed factory deep-clones complete seed/reference data.
- First load writes one complete seed; later loads use the stored snapshot.
- Export produces a current-version JSON envelope and excludes auth stores.
- Import validates fully before committing and appends a local import audit event.
- Reset explicitly replaces business data with a complete canonical seed, appends reset audit
  history, and preserves the active account's store identity fields (`name`, `vertical`, `province`,
  and `phone`).
- Corrupt/future persisted bytes are preserved byte-for-byte. The app loads a complete seed into
  protected memory with recovery metadata; ordinary commits remain memory-only and cannot overwrite
  the stored evidence.
- Only explicit valid import or reset may replace protected recovery bytes. If that explicit
  replacement cannot persist, recovery remains protected.
- If storage is blocked, the repository continues in memory; reload durability is not guaranteed.

### Legacy pre-scope adoption

- The old global `kitluy.partner.domain` snapshot is never deleted or rewritten by adoption.
- The first eligible authenticated account receives a byte-for-byte copy only when it has no scoped
  snapshot.
- `kitluy.partner.domain.legacy-owner.v1` durably records the one account allowed to adopt it.
- Later accounts cannot inherit it. An existing scoped snapshot remains authoritative while the owner
  marker is still consumed to prevent later cross-account adoption.

Repository tests cover protected recovery, explicit-replacement failure, future-version preservation,
guest/account isolation, one-owner legacy adoption, authoritative scoped snapshots, and the earlier
validation paths. File-picker/account-switch/reload behavior remains `NOT RUN` in a live browser.

## Remote model still required

A production Supabase/API design needs store/tenant IDs, foreign keys, uniqueness, server timestamps,
transaction/RPC boundaries, RLS, idempotency, conflict/realtime policy, soft-delete/retention,
tamper-resistant audit, backup/restore, and a local-data migration plan. The nested browser snapshot is
an application model, not a final SQL schema.
